# 权限获取引导功能文档

## 功能目标

权限页帮助用户完成 StillLoop 运行专注判断所需的 macOS 权限。当前主要权限为：

- 屏幕录制。
- 摄像头。

权限页同时说明隐私边界：StillLoop 不保存截图或摄像头画面，权限只用于本机专注判断。

## 页面入口

用户会在以下路径进入权限页：

- 首次设置流程。
- 点击顶部“缺少权限”问题入口。
- 开始专注时发现权限不足。
- macOS 权限状态变化后用户主动回到权限页。

## 页面内容

权限页标题为“权限获取引导”。页面说明：

“StillLoop 仅在本机读取必要的屏幕与摄像头状态，用于判断是否需要提醒；不会保存截图或摄像头画面。”

页面包含两个权限行：

- 屏幕录制。
- 摄像头。

每个权限行展示当前状态、修复指引和是否允许。

## 屏幕录制权限状态

屏幕录制通过 `CGPreflightScreenCaptureAccess()` 判断。

状态规则：

- 已允许：显示“已允许”，无额外操作。
- 当前不是 app bundle 运行：显示“开发运行模式无法注册到系统权限列表”，提示用 `scripts/run-app.sh` 启动应用包。
- 未生效：提示用户到“系统设置 > 隐私与安全性 > 录屏与系统录音”允许 StillLoop。如果已经开启但仍未生效，需要关闭后重新开启一次，并重启 StillLoop。

屏幕录制不能直接弹出授权框，因此主要动作是打开系统设置。

## 摄像头权限状态

摄像头通过 `AVCaptureDevice.authorizationStatus(for: .video)` 判断。

状态规则：

- authorized：显示“已允许”。
- notDetermined：显示“未请求”，继续后触发 macOS 摄像头授权窗口。
- denied：显示“已拒绝”，引导去系统设置摄像头页。
- restricted：显示“受限制”，引导检查系统限制。
- unknown：显示“未知”，引导用户在系统设置检查。

## 继续按钮行为

点击“继续”时，应用按以下顺序处理：

1. 如果摄像头状态是未请求，调用系统摄像头授权。
2. 如果屏幕录制未允许，尝试打开屏幕录制设置页。
3. 如果摄像头已拒绝或受限，尝试打开摄像头设置页。
4. 如果权限都满足，进入后续流程。

后续流程是模型准备页或任务输入主页，取决于用户是否已完成初始设置。

## 开始专注时的权限门槛

即使用户已经通过首次设置，开始专注仍会再次判断权限。决策顺序为：

1. 摄像头未请求：先请求摄像头授权。
2. 屏幕录制未允许：回到权限页并打开屏幕录制设置。
3. 摄像头已拒绝或受限：回到权限页并打开摄像头设置。
4. 权限满足：继续开始会话。

这样可以避免权限页状态与系统真实状态不一致。

## 系统设置打开策略

打开系统设置时，应用先尝试 macOS URL，再尝试 `open` 系统命令。成功后显示“已打开系统设置，请在系统设置窗口中完成授权。”失败时显示“无法自动打开系统设置，请按上方路径手动打开。”

## 隐私承诺

权限页表达的产品承诺必须与实际实现一致：

- 截图和摄像头帧用于本机评估。
- 原始截图和摄像头画面不写入磁盘。
- 专注摘要和评估事件本地保存。
- 只有开始专注后才启动采集循环。

## 测试关注点

- 开发运行模式下，屏幕录制权限提示必须指向 `scripts/run-app.sh`。
- 摄像头首次授权后应继续原本的开始专注流程。
- 权限拒绝时不能直接进入专注页。
- 从系统睡眠或锁屏恢复后，应刷新权限状态。
